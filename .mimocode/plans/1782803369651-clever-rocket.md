# 文件同步工具功能扩展计划

## 目标

在现有单向增量同步工具基础上，实现三项功能：
1. **增量去重索引** — `dedup` 命令支持增量扫描，避免每次全量重算哈希
2. **双向同步** — 两个相似文件夹的双向同步（两边有增删改，最终一致）
3. **双向同步的增量机制** — 复用增量索引，只处理变化文件

## 架构概览

```
                    ┌─────────────────┐
                    │  fileindex      │  ← 核心基础设施（新增）
                    │  (bbolt)        │
                    │  路径→状态映射   │
                    └────────┬────────┘
                             │
              ┌──────────────┼──────────────┐
              │              │              │
     ┌────────▼────────┐ ┌──▼──────────┐ ┌─▼──────────────┐
     │ dedup (改造)    │ │ bisync (新增)│ │ sync (现有)     │
     │ 增量去重        │ │ 双向同步     │ │ 单向同步(不变)   │
     └─────────────────┘ └─────────────┘ └────────────────┘
```

## Step 1: 文件状态索引 — `internal/fileindex`

**目的**: 提供可复用的增量文件状态追踪基础设施，dedup 和 bisync 共用。

### 设计

```
bbolt bucket: "filestates"
key:   文件相对路径 (string)
value: FileState{Size, Mtime, Hash}
```

**文件**: `internal/fileindex/fileindex.go`

```go
type FileState struct {
    Size   int64
    Mtime  time.Time
    Hash   string   // xxh3 objectKey, 空表示未计算
}

type FileIndex interface {
    Get(path string) (FileState, bool, error)
    Put(path string, s FileState) error
    Delete(path string) error
    Iterate(fn func(path string, s FileState) bool) error
    // 批量操作（单事务原子）
    ApplyBatch(puts map[string]FileState, deletes []string) error
    Close() error
}

func Open(path string) (FileIndex, error)
```

**关键**: 与现有 `internal/index` 类似但更轻量——不涉及 CAS object/RefCount，纯粹追踪文件状态用于增量检测。

### 变更检测逻辑

```go
// IsUnchanged 判断文件是否与索引记录一致
func IsUnchanged(indexed FileState, current Size, current Mtime) bool {
    return indexed.Size == current.Size && paths.MtimeClose(indexed.Mtime, current Mtime)
}

// 需要重算哈希的条件：
// 1. 文件不在索引中（新增）
// 2. size 或 mtime 变化（内容可能改变）
// 3. mtime 变但 size 不变（内容可能未变，但保守重算）
```

### 测试

- CRUD 操作
- 批量原子写入
- 并发安全

## Step 2: 增量去重 — 改造 `internal/dedup`

**目的**: `dedup` 命令第二次运行跳过未变文件，只重算变化文件的哈希。

### 现有流程（每次全量）

```
scanner.Scan → 按size分组 → 并发算哈希 → 按(size,hash)分组 → 去重
```

### 改造后流程

```
1. scanner.Scan → 拿到所有文件的 (path, size, mtime)
2. 打开 dedup.db (fileindex)
3. 增量检测:
   - 新文件 (不在index) → 需要算hash
   - 已删除 (在index但不在disk) → 从index删除
   - size/mtime变化 → 重算hash
   - 未变化 → 保留index中的hash
4. 从index中按 (size, hash) 分组，找出重复组
5. 执行去重 (硬链接)
6. 更新 index (写入新文件状态，删除已删文件)
```

### 文件变更

- `internal/dedup/dedup.go` — 重构 `Run` 方法，增加 index 参数
- `internal/dedup/dedup_test.go` — 增量场景测试

### 命令行变更

```bash
# 新增 --index 参数（可选，默认在扫描目录同级创建 .dedup-index.db）
filesync dedup <目录> [--index PATH] [--dry-run] [--readonly] [--exclude PATTERN]...
```

### 边界情况

- 索引文件不存在 → 全量扫描（兼容首次使用）
- 文件被外部修改但 mtime 未变 → 保守策略：size 不变时仍信任 mtime（与 sync 一致）
- 硬链接文件的特殊处理：已是硬链接的文件自动跳过（现有逻辑不变）

## Step 3: 双向同步 — `internal/bisync`

**目的**: 两个相似文件夹的双向同步，检测两边变化并合并。

### 配置结构

```yaml
# config.yaml 新增 bisync 模式
mode: bisync          # 默认 sync（单向），可选 bisync

bisync:
  left: "D:\\Project\\Go_project"    # 左端目录
  right: "E:\\Backup\\Go_project"    # 右端目录
  conflict: keep-both    # 冲突策略: keep-both / left-wins / right-wins / newer-wins
  workers: 8
  verify: true
  exclude:
    - "**/.git/**"
    - "**/node_modules/**"
```

### 状态追踪

使用 `fileindex`，**两端各存一份索引**（冗余容灾，任一端丢失可从另一端重建）：

- 左端: `left/.bisync-index.db`
- 右端: `right/.bisync-index.db`

每个索引存储**对方的文件状态**（用于下次对比检测变化）：
- 左端索引: `relPath → FileState`（记录右端上次同步后的状态）
- 右端索引: `relPath → FileState`（记录左端上次同步后的状态）

同步流程：
1. 读取两端索引（若缺失则视为首次全量同步）
2. 扫描两端当前文件
3. 对比检测变化
4. 执行同步
5. 更新两端索引（记录对方最新状态）

重建逻辑：若一端索引丢失，从另一端索引反推 + 全量扫描该端重建。

### 变更检测算法

```
扫描左端 → leftFiles map[path]{size, mtime}
扫描右端 → rightFiles map[path]{size, mtime}
读索引 → leftIndexed map[path]{size, mtime, hash}
          rightIndexed map[path]{size, mtime, hash}

对每个 path:
  inLeft  = leftFiles[path]  != nil
  inRight = rightFiles[path] != nil
  leftChanged  = inLeft  && !IsUnchanged(leftIndexed[path],  leftFiles[path])
  rightChanged = inRight && !IsUnchanged(rightIndexed[path], rightFiles[path])
  leftGone  = !inLeft  && leftIndexed[path] != nil
  rightGone = !inRight && rightIndexed[path] != nil

  决策矩阵:
  ┌─────────────┬──────────────┬──────────────────────────────┐
  │ 左端        │ 右端          │ 动作                          │
  ├─────────────┼──────────────┼──────────────────────────────┤
  │ 新增/变化    │ 无变化        │ 复制 left → right             │
  │ 无变化       │ 新增/变化     │ 复制 right → left             │
  │ 新增/变化    │ 新增/变化     │ 冲突 → 按 conflict 策略处理    │
  │ 已删除       │ 无变化        │ 从 right 删除                  │
  │ 无变化       │ 已删除        │ 从 left 删除                   │
  │ 已删除       │ 已删除        │ 从索引移除                     │
  │ 已删除       │ 新增/变化     │ 冲突 → 按 conflict 策略处理    │
  └─────────────┴──────────────┴──────────────────────────────┘
```

### 冲突处理

- **keep-both**: 两端都保留，冲突文件重命名加 `.left`/`.right` 后缀
- **left-wins**: 左端覆盖右端
- **right-wins**: 右端覆盖左端
- **newer-wins**: mtime 更新的一端覆盖另一端（需先算哈希确认内容真的不同）

### 文件清单

- `internal/bisync/bisync.go` — 主编排器
- `internal/bisync/bisync_test.go` — 单元测试
- `internal/config/config.go` — 新增 BisyncConfig 结构

## Step 4: CLI 与配置集成

**文件**: `main.go`, `internal/config/config.go`

### 新增命令

```bash
filesync bisync [--config FILE] [--dry-run] [--workers N] [--verify | --no-verify]
filesync bisync --left DIR --right DIR  # 临时指定，不依赖 config
```

### 配置变更

```go
type Config struct {
    // 现有字段不变...
    TargetRoot string          `yaml:"target_root"`
    Workers    int             `yaml:"workers"`
    Verify     *bool           `yaml:"verify"`
    Sources    []SourceMapping `yaml:"sources"`
    Exclude    []string        `yaml:"exclude"`

    // 新增
    Mode    string      `yaml:"mode"`     // "sync" (默认) | "bisync"
    Bisync  *BisyncConfig `yaml:"bisync"`
}

type BisyncConfig struct {
    Left     string   `yaml:"left"`
    Right    string   `yaml:"right"`
    Conflict string   `yaml:"conflict"`  // keep-both / left-wins / right-wins / newer-wins
}
```
